<br>

<br>

<br>

가까운 지인의 물음으로 시작된 이 프로젝트는 개발이 하고싶어서 근질근질하던 나에게는 가뭄의 단비 같았다.

때마침, 아주 친하게 지내고 있는 이전 동료에게 바이브 코딩을 경험해 보라며 Claude Pro 버전을 일주일간 무료로 사용할 수 있는 권한을 줬는데, 사용해보니 너무 신세계가 아닌가!

## 일단 거래소부터

지인의 부탁은 코인 자동매매 시스템이 있으면 좋겠다는 요청이었다.

우선 코인 거래 사이트에서 API를 제공하는지부터 확인해야 했다. 지인이 알려준 사이트는 두 개가 있었는데 **'주맥스'** 그리고 **'바이비트'** 라는 사이트였다.

<br>

둘 다 해외 사이트인데, 국내 거래소인 업비트는 안 되냐고 물어봤더니 Long, Short 투자가 불가능해서 안 된다고 했다.

...대체 Long, Short이 뭐지?!

어쨌든 의문을 뒤로 하고, '주맥스'와 '바이비트' 중 **바이비트 거래소의 API 문서가 더 보기 편해서** 해당 사이트의 API를 이용해 코인 자동매매 시스템을 만들기로 가닥을 잡았다.

<br>
<br>

## 공부부터 해야겠다

API 연동 방향은 잡았는데, 문제가 있었다.

나는 코인에 대해서 하나도 모르고, 그와 비슷한 주식의 "주"도 몰랐다. 레버리지가 어떻고, Long이 어떻고, Short이 어떻고... 용어를 익히고 코인 시장이 어떻게 돌아가는지부터 배워야 했다.

<br>

| 용어 | 설명 |
|------|------|
| Long | 가격이 오를 것을 예상하고 매수하는 포지션. 오르면 수익, 내리면 손실 |
| Short | 가격이 내릴 것을 예상하고 매도하는 포지션. 내리면 수익, 오르면 손실 |
| 레버리지 | 내 자본보다 더 큰 금액으로 거래하는 방식. 수익도 크지만 손실도 크다 |
| 청산 | 레버리지 거래에서 손실이 일정 수준을 넘으면 포지션이 강제로 종료되는 것 |
| 캔들 | 특정 시간 동안의 시가, 종가, 고가, 저가를 하나의 막대로 표현한 차트 |
| 펀딩비 | 선물 거래에서 Long과 Short 포지션 보유자 간에 주기적으로 교환되는 비용 |

<br>

이것도 빙산의 일각이고 실제로는 훨씬 더 많은 개념들이 있었다. 공부하면 할수록 "이걸 내가 진짜 만들 수 있나?" 싶었지만...

뭐, 일단 해보는 거지.

<br>
<br>

## 지인과의 미팅 — 근데 지인도 몰랐다

공부를 어느 정도 했으니 이제 본격적으로 뭘 만들어야 하는지 얘기해보자 싶었다. 그래서 지인과 미팅을 잡았다.

근데 여기서 예상 못 한 상황이 펼쳐졌다.

**지인도 자기가 뭘 원하는지 정확히 몰랐다.**

"자동으로 사고팔렸으면 좋겠어"는 알겠는데, 어떤 기준으로? 어떤 코인을? 손실이 나면 어떻게? 이런 세부적인 질문을 던지면 "음... 그건 잘 모르겠는데"가 돌아왔다. 생각해보면 당연한 거였다. 개발에 대해서도, 코인 자동매매 전략에 대해서도 잘 모르는 분이니, 막연하게 필요하다는 건 알지만 구체적인 그림은 없었던 것이다.

<br>
<br>

## 그래서 선택한 방법론 — 프로토타입 모델

이 상황에서 개발 방법론을 고민했다.

요구사항이 처음부터 명확하다면 폭포수 모델처럼 순서대로 쭉 진행하면 된다. 하지만 지금 상황은 달랐다. 요구사항 자체가 미팅을 거듭하면서 점점 구체화되는 형태였기 때문에, **프로토타입 모델**이 맞겠다고 판단했다.

<br>

프로토타입 모델은 이런 흐름으로 진행된다.

```
요구사항 파악 → 프로토타입 제작 → 사용자 평가 → 요구사항 구체화 → 반복
```

<br>

완성된 결과물을 한 번에 만드는 게 아니라, 일단 대략적인 프로토타입을 만들어 보여주고 "이런 거 맞아요?" 하고 확인받는 식이다. 그 피드백을 반영해서 다시 구체화하고, 또 확인받고. 이 과정을 반복하면서 요구사항이 점점 선명해진다.

눈앞에 뭔가를 보여줘야 "아, 이건 이렇게 바꾸고 싶어"라는 말이 나오는 법이니까.

<br>
<br>

## 요구사항 정의서(SRS)

미팅을 여러 번 반복하면서 나온 내용들을 그때그때 요구사항 정의서에 쌓아갔다. 한 번에 다 나온 게 아니라, 프로토타입을 보고 피드백을 주고, 그게 반영되면 또 새로운 요구사항이 나오고, 이런 식으로 점점 채워졌다.

요구사항 정의서는 건축으로 치면 설계도 같은 것이다. 이게 잘 만들어져야 이후에 흔들리지 않는다. 반대로 이게 흐릿하면 개발하면서 계속 방향을 잃게 된다.

<br>

**📌 사용자 관리**
- 로그인을 필수로 해야 한다
- 계정이 없다면 회원가입을 통해 계정을 생성할 수 있어야 한다

**📌 대시보드**
- 코인 목록이 노출되어야 하며, 거래량 기준 내림차순으로 정렬되어야 한다
- 특정 코인을 클릭하면 상세 차트를 확인할 수 있어야 한다
- 어느 화면에서든 현재 선물 지갑 잔액을 확인할 수 있어야 한다
- 등록된 큐 목록을 한눈에 확인할 수 있어야 한다

**📌 자동매매 시스템 (큐)**
- 코인별로 자동매매 큐를 설정할 수 있어야 한다
- 큐는 여러 단계로 구성되며, 각 단계마다 Long/Short 패턴을 정의할 수 있어야 한다
- 모든 거래는 시장가 / 아이솔레이트 방식으로 체결된다
- 단계별로 레버리지 배율을 설정할 수 있어야 한다
- 큐는 활성화/비활성화가 가능해야 한다

**📌 상세 차트**
- 활성화된 큐의 현재 진행 단계 및 패턴을 시각적으로 확인할 수 있어야 한다
- 큐 활성화 시점을 기준으로 손익 상태를 확인할 수 있어야 한다
- 호가창과 패턴 관리 UI를 함께 제공해야 한다

**📌 거래 내역**
- 패턴에 의해 체결된 거래 이력이 모두 기록되어야 한다
- 거래가 실패했을 때도 시도 내역이 남아야 한다

<br>

정리하고 나니 생각보다 꽤 많은 기능이 들어가야 한다는 게 보였다. 그리고 이 단계를 제대로 해뒀더니, 이후 개발에서 "이걸 만들어야 하나 말아야 하나" 같은 고민이 확실히 줄었다.

<br>
<br>

## 엔터티 도출

요구사항이 정리되고 나면 자연스럽게 **어떤 데이터가 필요한지** 보이기 시작한다.

이걸 엔터티로 뽑아냈다.

<br>

```
USERS            → 사용자 정보
PATTERN_QUEUE    → 패턴 큐
PATTERN_STEP     → 패턴 단계
PATTERN          → 패턴
PATTERN_BLOCK    → 패턴 블록
TRADE_HISTORY    → 패턴 단계 히스토리 - 스냅샷 형태
PATTERN_HISTORY  → 패턴 단계 히스토리 - 스냅샷 형태
```

<br>

엔터티 간의 관계를 조금만 가져와보면 다음과 같다.

- `User`는 여러 `PATTERN_QUEUE`를 가질 수 있다 (1:N)
- `PATTERN_QUEUE`는 여러 `PATTERN_STEP`을 가질 수 있다 (1:N)

<br>
<br>
