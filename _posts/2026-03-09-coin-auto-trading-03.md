# 아키텍처 구상하기

<br>

[2편 보러가기](https://mhs442.github.io/2026-02-27/coin-auto-trading-02)

전편에서 간단한 DB 설계까지 해보았다. 이번에는 실제 구현에 앞서 기술 스택과 이에 따른 아키텍처를 선정할 차례이다.

<br>

## 인프라부터 생각해볼까?

<br>

기술 스택을 정하기 전에, 이 프로그램이 어디에서 어떻게 돌아갈지를 먼저 생각해봤다.

<br>

1. 프로그램을 부탁한 지인에게 사용하지 않는 개인 PC가 있어 해당 PC를 서버로 활용할 수 있다.
2. 집에서만 개인적으로 사용할 예정이라고 한다. 외부에서 접근할 일이 없다.

<br>

> 그래서 내린 결론. **온프레미스** 형식으로 설치하도록 하자!

<br>

혼자 사용할 것이고 트래픽도 본인 한 명분만 감당하면 되니, 한 대의 PC에 모두 묶어서 설치하는 게 가장 간단했다.

<br>

다만 유지보수가 걸렸다. 버그 수정이나 업데이트가 필요하면 직접 찾아가서 작업해야 한다는 점이.. 그렇다고 클라우드에 올리자니 서버 이용 요금이 발생한다. 실제로 예전에 AWS EC2를 띄워놓고 깜빡 잊었다가 요금이 청구된 적이 있어서.. 개인 프로젝트에 클라우드는 부담스러웠다.

<br>

근데 곰곰이 생각해보니, 꼭 직접 찾아가야만 하나? 어차피 지인 PC에 OS가 깔려 있고 네트워크에 연결되어 있으니, 같은 네트워크 안에서 SSH로 원격 접속하면 되는 거 아닌가. 외부에서 접근이 필요해지더라도 VPN이나 SSH 터널링으로 충분히 해결할 수 있다. 생각해보니 별거 아니었다.

<br>

요새 유행하는 MSA니 분산 환경이니 하는 아키텍처도 잠깐 스쳐 지나갔지만, 사용자 한 명짜리 프로젝트에 그건 명백한 오버엔지니어링이다. 깔끔하게 **모놀리식**으로 가되, 배포는 **Docker**를 활용하기로 했다.

<br>

애플리케이션과 MariaDB를 각각 Docker 이미지로 빌드하고, 쉘 스크립트에 `docker run -p` 명령어를 넣어두면 스크립트 하나로 실행이 끝난다. 지인 PC에 가서 세팅할 때도 Docker만 설치되어 있으면 되니까 환경 구성이 훨씬 간단해진다.

<br>

### 그런데.. 서버가 죽으면?

<br>

한 가지 더 고민한 부분이 있었다. 이 프로그램은 **코인 자동매매 시스템**이다. 24시간 돌아가야 하는 프로그램인데, 서버 PC가 꺼지거나 네트워크가 끊기면 자동매매가 멈춘다. 시장이 급변하는 시점에 시스템이 죽어있으면 그게 곧 손실로 이어질 수 있다.

<br>

그래서 서버의 생존 여부를 감시하는 방법을 고민했다. 서버 안에서 자기 자신을 감시하는 건 의미가 없다. 서버가 죽으면 감시 프로세스도 같이 죽으니까. **감시하는 주체가 서버 바깥에 있어야 한다.**

<br>

마침 내 개발 PC가 상시 가동 중이다. Discord 봇 서버로 활용하고 있어서 사실상 24시간 켜져 있는 상태니까, 이 PC에서 지인의 서버를 주기적으로 감시하면 되겠다는 생각이 들었다.

<br>

구조는 간단하다. 지인 서버에 Health Check API를 하나 만들어두고, 내 개발 PC에서 주기적으로 호출한다. 응답이 없으면 SMTP 프로토콜을 활용해서 메일로 알림을 보내는 방식이다.

<br>

```
[내 개발 PC] --(주기적으로 Health Check 호출)--> [지인 서버]
                  │
            응답 없음?
                  │
        [Gmail SMTP] --> 사용자에게 메일 알림
```

<br>

SMTP를 직접 다뤄보고 싶은 욕심도 있었고, 외부 서비스에 의존하지 않고 직접 만들어보는 게 학습에도 도움이 될 것 같았다. 이 부분은 이후 개발 과정에서 자세히 다뤄볼 예정이다.

<br>

### 확장은 어디까지 고려할까?

<br>

지금은 애플리케이션과 MariaDB, 컨테이너 2개로 충분하다. 쉘 스크립트에 `docker run` 명령어를 넣어두면 관리에 불편함이 없다.

<br>

하지만 나중에 서비스가 커질 가능성을 아예 무시할 수는 없었다. 예를 들어, ByBit에서 가져오는 시세 데이터를 매번 API로 조회하는 건 비효율적이다. 자주 조회되는 데이터는 캐시에 올려두는 게 맞고, 그러면 자연스럽게 **Redis** 같은 인메모리 저장소가 필요해진다. 인증 방식을 JWT로 전환하게 되면 Refresh Token 관리에도 Redis가 필요할 수 있고.

<br>

컨테이너가 3개, 4개로 늘어나면 각각의 `docker run` 명령어를 쉘 스크립트에 나열하는 것도 한계가 온다. 컨테이너 간 네트워크 설정이나 실행 순서 의존성까지 관리해야 하니까. 그때가 되면 **Docker Compose**를 도입해서 `docker-compose up` 한 줄로 전체 환경을 올리는 방식으로 전환하면 될 것이다.

<br>

```
현재:  쉘 스크립트 (docker run)
        ├─ 애플리케이션 컨테이너
        └─ MariaDB 컨테이너

추후:  Docker Compose (docker-compose up)
        ├─ 애플리케이션 컨테이너
        ├─ MariaDB 컨테이너
        └─ Redis 컨테이너
```

<br>

당장은 필요 없지만, 이런 확장 경로가 있다는 걸 머릿속에 두고 설계하는 것과 아닌 것은 차이가 크다. 우선은 현재 필요한 부분에 집중하되, 나중에 전환하기 어렵지 않도록 구조만 열어두기로 했다.

<br>

## 기술 스택 선정

우선 내가 사용이 가능한 언어들로 작업을 하기로 했다.

<br>

### Backend

<br>

**1. Java 17**

장기 지원 버전인 Java 21이 있지만 기능상으로 크게 차이나는 부분은 없어 기존에 익숙하게 쓰던 Java 버전을 쓰기로 했다. 
다만, 21 버전에서 추가된 가상 스레드는 경량 스레드로써 기존보다 훨씬 적은 리소스를 사용한다고 한다. 동시성을 위해 나중에 Java 21로의 버전 변경을 염두에 두고 작업을 할 것이다.

<br>

**2. Spring Boot 3.4.2**

이건 고민할 것도 없었다. Java 기반 웹 애플리케이션이라면 사실상 표준이니까. 최신 버전을 쓰면서 Spring Security, Spring Data JPA 등 필요한 모듈을 자유롭게 가져다 쓸 수 있다는 게 큰 장점이다.

<br>

**3. Spring Data JPA (Hibernate)**

ORM은 JPA를 사용하기로 했다. 
토이 프로젝트 수준에서 MyBatis까지 쓸 필요는 없었고, 엔터티 설계도 이미 끝난 상태라 JPA가 더 깔끔하게 떨어졌다.
게다가 빠르게 개발하고 바로바로 보여줘야 하기 때문에 Spring Data JPA가 가장 적합했다.

<br>

**4. MariaDB**

데이터베이스는 MariaDB를 선택했다. 내가 사용하고 있는 개인 서버에 개발용으로 이미 설치되어 있는 데이터베이스이기도 하고...
MySQL과 호환성이 좋으면서도 오픈소스이고, 개인 프로젝트에서 쓰기에 부담이 없다. 테스트 환경에서는 H2 인메모리 DB를 사용해서 빠르게 돌릴 수 있도록 했다.

<br>

**5. Spring Cloud OpenFeign**

ByBit API 호출을 위한 HTTP 클라이언트로 OpenFeign을 선택했다. RestTemplate이나 WebClient를 직접 쓸 수도 있었지만, 인터페이스만 정의하면 구현체를 알아서 만들어주는 선언적 방식으로 간단하니까.

<br>

**6. Spring WebSocket**

ByBit에서 실시간 시세 데이터를 받아오기 위해 WebSocket을 사용하기로 했다. REST API로 매초 폴링하는 건 비효율적이니까. WebSocket으로 연결해두면 시세가 변할 때마다 서버에서 데이터를 밀어주니 실시간성도 확보되고 서버 부하도 줄일 수 있다.

<br>

연결이 끊어졌을 때를 대비해서 자동 재연결 로직도 넣었다. 지수 백오프(Exponential Backoff) 방식으로 1초 → 2초 → 4초 → ... → 최대 30초 간격으로 재시도하도록 설계했다. 무작정 바로바로 재연결하면 서버에 부담을 주니까.

<br>

**7. Spring Security**

로그인 기능이 필요하니 Spring Security를 도입했다. 폼 기반 로그인 방식으로, 휴대폰 번호를 아이디 대신 사용한다. 비밀번호는 당연히 **BCrypt**로 해싱해서 저장하고, ByBit API Key는 **AES-256 GCM** 방식으로 암호화해서 DB에 넣는다. API Key가 평문으로 저장되면.. 생각만 해도 아찔하다.

<br>

### Frontend

<br>

프론트엔드는 최대한 심플하게 가기로 했다. React나 Vue 같은 프레임워크를 쓸까 잠깐 고민했는데, 이 프로젝트에서 프론트엔드가 차지하는 비중을 생각하면 오버엔지니어링이라고 판단했다.

<br>

**1. Thymeleaf**

서버 사이드 렌더링으로 Thymeleaf를 선택했다. Spring Boot와의 궁합이 좋고, 별도의 빌드 파이프라인 없이 HTML 파일만으로 화면을 구성할 수 있다는 게 장점이다.

<br>

**2. jQuery**

DOM 조작이나 AJAX 호출 등 동적인 처리를 위해 jQuery를 사용했다. 순수 JS만으로도 가능하지만, 익숙한 도구를 쓰는 게 생산성 면에서 나았다.

<br>

### 테스트

<br>

2편에서도 이야기했지만 Claude에게 TDD 방식으로 개발하라고 지시를 해뒀다. 테스트 전략은 다음과 같다.

<br>

| 레이어 | 테스트 방식 | 비고 |
|--------|-----------|------|
| Controller | 통합 테스트 | MockMvc 활용 |
| Service | 단위 테스트 | Mockito로 의존성 모킹 |
| Repository | 통합 테스트 | H2 인메모리 DB |
| Client (OpenFeign) | 통합 테스트 | WireMock으로 API 응답 모의 |

<br>

특히 ByBit API 호출 부분은 **WireMock**을 이용해서 모의 응답을 만들어 테스트한다. 실제 ByBit 서버에 요청을 날리면서 테스트할 수는 없으니까. 모의 응답 데이터는 JSON 파일로 관리하고, `test/resources/__files/` 디렉터리 하위에 `클라이언트명_메서드명.json` 형태로 저장한다.

<br>

### 빌드 도구

<br>

**Gradle**을 사용했다. Maven도 나쁘지 않지만, 빌드 스크립트가 XML 기반이라 가독성이 떨어진다. Gradle은 Groovy DSL로 작성할 수 있어서 의존성 관리가 직관적이다.

<br>
